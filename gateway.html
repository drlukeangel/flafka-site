How It Works

Replication in the data path

No separate replication service. No per-GB Confluent fees. The gateway already sits between your producers and Kafka — dual-write adds replication in that same hop.

Producers write to the gateway

Your producers connect to the Flafka gateway endpoint as if it were a Kafka broker. Zero changes to your application code — same bootstrap address pattern, same client libraries.

0 application code changes
2

Gateway clones every ProduceRequest

The gateway forwards ProduceRequests to the primary cluster synchronously, then asynchronously clones the same frame to the secondary cluster. Per-partition bounded queues prevent secondary slowness from affecting the primary path.

<1ms overhead on primary path
3

Consumers failover to secondary

If the primary cluster fails, consumers failover to the secondary. Phase 1: timestamp-based consumer reset (<1 min RTO). Phase 2: offset translation for near-zero RPO and <10 second RTO.

RTO <10s (Phase 2)

Replication Roadmap

Four phases to full active-active

Phase 1 is available in beta today. Each phase builds on the last without breaking changes.

1
Phase 1: Fire-and-Forget Cloning
Beta Available
  • Real-time ProduceRequest cloning to secondary cluster
  • <1ms overhead on the primary path (circuit breaker on secondary)
  • Topic auto-creation mirroring primary topic configuration and partition count
  • Per-topic replication lag dashboard in Flafka Ops UI
  • RPO: non-zero (fire-and-forget; data in async queue at crash time is at risk)
2
Phase 2: Reconciliation Consumer
Q3 2026
  • Reconciliation consumer replays any gap from primary log after recovery
  • Offset mapping and translation for consumer failover
  • RTO <10 seconds with offset translation
  • RPO: near-zero (reconciliation closes the async gap)
3
Phase 3: EOS / Idempotent Deduplication
Q4 2026
  • Exactly-once semantics on the replication path
  • Idempotent deduplication prevents duplicate writes on failover
  • Full transactional producer support in dual-write mode
4
Phase 4: Active-Active Bidirectional Replication
2027
  • Both clusters accept writes simultaneously
  • Conflict resolution policies (last-write-wins, custom)
  • Circular replication prevention via message provenance headers
Operational Visibility

Replication Monitoring & Alerting

Real-time lag delta, per-partition ISR status, circuit breaker state, and auto-alerts for replication events — all in the same UI where you manage your topics.

Real-Time Lag Dashboard

  • Per-topic replication lag (ms)

    Sparklines show which topics are keeping up and which are falling behind in real time.

  • Per-partition ISR status

    ● green: ISR healthy  |  ● yellow: ISR shrinking  |  ● red: ISR < min.insync.replicas

  • Secondary cluster health

    Circuit breaker state: Closed (healthy), Open (tripped), HalfOpen (recovering).

  • Buffer utilization gauge

    Shows how much overflow buffer is consumed — early warning before backpressure impacts producers.

Health Thresholds & Auto-Alerts

  • Configurable lag threshold — alert when replication lag exceeds N ms per topic
  • ISR shrinkage alert — fires when a partition ISR count drops below your configured minimum
  • Circuit breaker trip alert — fires when the secondary cluster experiences repeated failures
  • Flexible routing — alerts route to Slack, PagerDuty, or any webhook target in your Alert Routing settings
  • Pre-built Grafana dashboard — dual-write-grafana-dashboard.json, 10 panels, Prometheus-compatible, download and import in minutes

Prometheus Integration

Point your Prometheus scrape config at the Flafka metrics endpoint. All metrics are labelled by topic and partition for fine-grained alerting.

  • replication_lag_ms — histogram per topic-partition
  • circuit_breaker_state — gauge: 0=Closed, 1=Open, 2=HalfOpen
  • buffer_bytes — current overflow buffer fill level
  • buffer_overflow_total — cumulative overflow events counter

Compatible with any Prometheus-compatible scrape target: Grafana Cloud, Datadog, VictoriaMetrics.

FAQ

Common questions

Does this require changes to our producer application code? +
No. Producers connect to the Flafka gateway endpoint as if it were a Kafka broker. The gateway handles replication transparently — no client code changes, no library updates, no configuration changes on the producer side. If you can change a bootstrap address in your config, you can enable dual-write replication.
What is the RPO in Phase 1? +
Non-zero. Data in the async send pipeline at crash time is at risk of loss. This is the honest trade-off of the fire-and-forget approach. If your workload requires sub-second RPO today, Confluent Cluster Linking currently has an advantage. Phase 2 (reconciliation consumer) reduces RPO to near-zero by replaying any gap from the primary cluster log after recovery. Phase 2 ships Q3 2026.
Which Kafka clusters are supported? +
Confluent Cloud, AWS MSK, Redpanda, and self-managed Apache Kafka. Any Kafka-compatible cluster that speaks the Apache Kafka wire protocol is supported — as both primary and secondary. This means you can replicate from Confluent Cloud to MSK, MSK to MSK, or any other combination Cluster Linking cannot support.
Is this a replacement for Cluster Linking? +
Yes, for most use cases. Cluster Linking has some advantages — offset preservation from day one, deep Confluent Cloud integration. Flafka Gateway closes the offset preservation gap in Phase 2. For multi-cloud or mixed-cluster DR (e.g., Confluent Cloud to MSK), Flafka Gateway is the only option: Cluster Linking only replicates between Confluent Cloud clusters.
