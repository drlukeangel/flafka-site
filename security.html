Compliance Frameworks

7 frameworks, one platform

Flafka features mapped to the compliance requirements your enterprise needs.

💳
PCI-DSS

Resource-level RBAC with glob patterns restricts access to PII topics by pattern (e.g., pii-*). Topic-level read/write separation. Full audit trail for all access.

  • Req 7: Least-privilege access via resource-level RBAC
  • Req 8: Unique user identification with bcrypt-hashed passwords
  • Req 10: Full audit trail with JSONL persistence and 30-day retention
📝
SOX Compliance

Complete audit trail covering session events (login/logout/refresh), ACL changes, role assignments, and resource pattern modifications. JSONL persistence with daily rotation.

  • Session audit: login, logout, token refresh events
  • ACL change tracking with before/after snapshots
  • Role assignment audit with actor attribution
  • 30-day retention with daily log rotation
🛡
SOC 2 Type II Readiness

RBAC enforcement middleware, permission-denied logging, bootstrap admin with lockout prevention, auth gate enforcement, and OIDC/SSO support.

  • CC6.1: Logical access controls with RBAC middleware
  • CC6.3: Role-based access with glob pattern matching
  • CC7.2: System monitoring via permission-denied logging
  • Bootstrap admin with lockout prevention
🏥
HIPAA Considerations

PII topic restrictions via resource patterns, audit trail completeness, and session tracking support HIPAA's access control and audit requirements.

  • Access control via resource patterns (e.g., phi-*)
  • Complete audit trail for all user actions
  • Session tracking with IP anonymization
  • Data masking roadmap (upcoming)

GDPR

IP anonymization in session audit (IPv4 last octet masked, IPv6 last 80 bits masked). User data export via analytics export API.

  • Art 25: Privacy by design with IP anonymization
  • Art 17: User data export via analytics API
  • Self-hosted — data never leaves your infrastructure
  • No external telemetry or data collection

  • Self-hosted deployment — deploy in any region

Identity & Authentication

Flafka as an OIDC Identity Provider

Flafka issues RS256-signed JWTs for authenticated users and service accounts — eliminating the need for a third-party IdP in dev and staging environments.

  • RS256-signed JWTs — Flafka issues RS256 JWTs for all authenticated users and service accounts
  • JWKS endpoint — available at /api/v1/auth/.well-known/jwks.json — configure your Kafka clients to use Flafka as the token authority
  • Gateway OAUTHBEARER Mode 2 — producers and consumers authenticate to Flafka Gateway using Flafka-issued tokens. No third-party IdP required for dev/staging environments
  • Production OIDC integration — integrate with your existing OIDC provider (Okta, Azure AD, Auth0, Keycloak) via Flafka's OIDC upstream configuration
  • SAML 2.0 — on roadmap. Contact enterprise sales →

OIDC Provider: Pro/Team and Enterprise. See pricing →

Conduktor and Redpanda Console both require an external identity provider. Flafka includes its own OIDC issuer — dev and staging environments have zero external IdP dependency. For production, integrate with Okta, Azure AD, Auth0, or Keycloak via Flafka's OIDC upstream configuration.
