1. Executive Summary

Apache Kafka powers the real-time data infrastructure for thousands of enterprises. As Kafka deployments grow, so do compliance requirements: who accessed which topic, when were ACLs changed, what SLOs are being met, and can we prove it to auditors?

Flafka is the only Kafka management platform with a dedicated compliance story. While competitors like Conduktor, AKHQ, Kafka UI, and Redpanda Console focus on operational features, Flafka ships 9 compliance building blocks that map directly to SOC 2 Trust Services Criteria.

This guide maps Flafka's shipped features to the five most relevant SOC 2 criteria, provides evidence types for each, and shows how to generate compliance artifacts your auditors need.

Why Kafka Management Needs Compliance Controls

Kafka topics often contain PII, financial transactions, and healthcare data. Without a management layer that enforces access controls, logs all actions, and monitors for anomalies, organizations face:

  • Unauthorized access to sensitive topics (PCI-DSS Req 7 violation)
  • No audit trail for configuration changes (SOX non-compliance)
  • No evidence of monitoring for SOC 2 auditors (CC7.2 gap)
  • Manual, error-prone compliance reporting

Flafka addresses all four gaps with features that are shipped, tested, and generating evidence today.

2. SOC 2 Trust Services Criteria Mapping

The following table maps the five most relevant SOC 2 Common Criteria to specific Flafka features, the sprint in which each feature shipped, and the type of evidence available for auditors.

CC6.1 Logical Access Controls RBAC with glob patterns, Bootstrap Admin, ACL templates S72, S71 RBAC Export
CC6.3 Role-Based Access Resource-level RBAC, permission-denied guidance, ACL CRUD S72, S71 Screenshot + Audit Log
CC7.2 System Monitoring SLO tracking, Prometheus export, connector monitoring, SLO compliance reports S67, S69, S79 SLO Report Export
CC7.3 Anomaly Detection & Response SLO webhook alerts, statement failure alerts, connector error alerts S74, S79 Webhook Logs

CC6.1 — Logical Access Controls

Flafka's RBAC system uses glob patterns (e.g., pii-*, prod-payments-*) to restrict access at the resource level. Users are assigned roles with specific read/write permissions on topic patterns, schema subjects, and connector names. The Bootstrap Admin feature prevents lockout during initial setup.

Evidence: Export the full RBAC configuration via Settings > Security > Export RBAC. The export includes all users, roles, resource patterns, and permission grants.

CC6.3 — Role-Based Access

ACL templates provide pre-configured role definitions (e.g., "Read-Only Analyst", "Stream Engineer", "Admin") that can be applied to new users. Permission-denied events are logged with actionable guidance, helping users request appropriate access rather than escalating to admin.

Evidence: Permission-denied log entries with user, resource, action, and timestamp. ACL change history in the audit trail.

CC7.2 — System Monitoring

SLO tracking monitors consumer lag, partition balance, and throughput against configurable thresholds. SLO compliance reports (shipped Sprint 79) provide exportable summaries showing compliance percentages over time. Prometheus export enables integration with existing monitoring infrastructure.

Evidence: SLO compliance report exports (JSON/CSV), Prometheus metrics endpoint, connector throughput charts.

CC7.3 — Anomaly Detection & Response

When SLOs are violated, Flafka sends webhook alerts to Slack, PagerDuty, or any HTTP endpoint. Statement failure alerts notify teams of Flink SQL errors. Connector error alerts detect task failures and unhealthy connectors.

Evidence: Webhook delivery logs, alert configuration exports, SLO violation history.

CC8.1 — Change Management

Every configuration change is recorded in a JSONL audit trail with daily rotation and 30-day retention. This covers: connector config changes, session events (login/logout/refresh), ACL modifications, role assignments, and resource pattern updates.

Evidence: JSONL audit log files with timestamps, actor IDs, before/after snapshots, and action types.

4. How to Generate Compliance Evidence

When your auditor requests compliance evidence, Flafka provides four export mechanisms:

4.1 RBAC Configuration Export

Navigate to Settings > Security > Export RBAC. This generates a JSON file containing all users, roles, resource patterns, and permission grants. Useful for CC6.1 and CC6.3 evidence.

4.2 SLO Compliance Reports

Navigate to SLO Dashboard > Compliance Reports. Export reports as JSON or CSV showing SLO compliance percentages, violation counts, and error budget consumption over configurable time ranges. Useful for CC7.2 evidence.

4.3 Data Quality Violation Export

Navigate to Data Quality > Violations > Export. Download violation history with rule definitions, affected topics, and severity levels. Useful for PI1.3 (data integrity) evidence.

4.4 Audit Log Files

Audit logs are stored as JSONL files with daily rotation in the configured audit directory. Each entry includes timestamp, actor ID, action type, resource, and before/after state for mutations. 30-day retention by default. Useful for CC8.1 evidence.

Example Audit Log Entry

{
  "timestamp": "2026-03-22T14:32:01Z",
  "actor": "admin@company.com",
  "action": "acl.update",
  "resource": "topic:payments-*",
  "before": { "permission": "read" },
  "after": { "permission": "read,write" },
  "ip": "192.168.1.xxx"
}

5. Getting Started

5.1 Deploy in One Command

docker run -d -p 8080:8080 \
  -e FLAFKA_CONFLUENT_CLOUD_API_KEY=your-key \
  -e FLAFKA_CONFLUENT_CLOUD_API_SECRET=your-secret \
  -e FLAFKA_CONFLUENT_CLUSTER_ID=your-cluster-id \
  flafka/flafka:latest

5.2 First SLO in 5 Minutes

  1. Open Flafka at http://localhost:8080
  2. Navigate to SLO Dashboard
  3. Click Create SLO
  4. Select a topic, set a lag threshold (e.g., 1000 messages), and target (e.g., 99.5%)
  5. Enable webhook alerts to your Slack channel
  6. Flafka begins tracking immediately — no restart required

5.3 Enable RBAC

  1. Navigate to Settings > Security
  2. Enable RBAC enforcement
  3. Create roles using glob patterns (e.g., prod-* for production topics)
  4. Assign roles to users
  5. Export the RBAC configuration for your audit file
