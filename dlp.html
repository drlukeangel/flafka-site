The Problem

Kafka Was Not Built for PII

  • Manual regex rules miss evolving PII patterns (new card formats, email structures, international ID variations)
  • Schema-only validation cannot detect PII embedded in free-form string fields — JSON blobs, log messages, user-generated content
  • Compliance teams need field-level audit trails, not just topic-level access controls

Without automated PII detection in the data path, sensitive data silently flows into Kafka topics — discovered only after a compliance audit finds it months later. By then, it may have replicated to 12 downstream consumers.

What We Detect

13 PII Types. Regex + ML.

Each type uses a combination of deterministic regex patterns (for known formats) and a trained ML classifier (for contextual and evolving variants). Detection runs entirely on-premises — no data leaves your network.

Credit Card Numbers
Regex + Luhn check for all major card networks (Visa, MC, Amex, Discover, UnionPay).
👤
Social Security Numbers (US)
Pattern + context detection for formatted and unformatted US SSNs.
🏴
National ID Numbers

Email Addresses
RFC-compliant email detection including subdomain and plus-addressing variants.

Phone Numbers (International)
E.164 and local formats for 60+ countries with ML disambiguation from non-phone numbers.

Physical Addresses
ML-based address component detection — street, city, postal code, country combinations.

Full Names (Contextual)

Date of Birth

Biometric Identifiers
