Sprint 104
New Feature
June 24, 2026

Consumer Failover API, Header Injection, and Consumer Group Offset Translation

  • Automated Consumer Failover API (WS1) — POST /api/v1/replication/{domain}/failover triggers consumer group failover with translated offsets; batch offset map lookup for all partitions in a consumer group; failover state machine (pending → in_progress → completed → rolled_back); rollback API; failover audit log for compliance; RTO <10 seconds
  • Optional Header Injection (WS2) — opt-in per topic; injects __flafka_primary_offset, __flafka_source_cluster, and __flafka_schema_id headers into replicated records; enables exact record-level correlation between primary and secondary clusters without consumer-side changes
  • Consumer Group Offset Translation (WS3) — periodic sync of consumer group committed offsets from primary to secondary; GET /api/v1/replication/{domain}/consumer-offsets/{group} returns translated offset positions; timestamp-based fallback for groups not in mapping table
Sprint 103
New Feature
June 10, 2026

Reconciliation Consumer, Offset Mapping, and Near-Zero RPO

  • Fire-and-Record Mode + Offset Mapping Table (WS1) — captures secondary ProduceResponse and builds in-memory offset mapping table (topic, partition, primary_offset) → secondary_offset); bounded LRU map (default 10M entries); periodic flush to __flafka_replication_offsets topic; offset map query API
  • Reconciliation Consumer (WS2) — background consumer that compares primary/secondary high-water marks; automatically fills replication gaps from primary log on secondary cluster recovery; RPO drops to near-zero (bounded by replication lag at failure moment, typically <5ms)
  • Overflow Buffer + Enhanced Monitoring (WS3) — bounded in-memory ring buffer (default 100 MB) holds failed replication requests; on secondary recovery, buffer drains; Grafana dashboard for replication lag, throughput, error rates, buffer utilization
Sprint 102
New Feature
May 27, 2026

Gateway Production Hardening: Topic Auto-Creation, ACL Mirroring, and UI Replication Status

  • Topic Auto-Creation on Secondary (WS1) — on first ProduceRequest to a replicated topic, automatically creates the topic on the secondary cluster matching primary config (partition count, replication factor, retention, compaction); existence cache prevents repeated Metadata API calls; partition count mismatch is logged as a warning
  • ACL Mirroring (WS1) — optional sync of topic-level ACLs from primary to secondary via CreateAcls API; configurable per-topic; default disabled
  • Performance Benchmarks + Latency Regression Suite (WS2) — criterion benchmarks for per-partition send queue throughput and frame clone latency; p99 latency regression test (primary path <1ms) runs on every CI push
  • Flafka UI Replication Status (WS3) — per-topic replication panel in the cluster overview: shows replication lag per partition, secondary cluster health, circuit breaker state (open/closed), and bytes replicated trend
Sprint 101
New Feature
May 13, 2026

Gateway Dual-Write Phase 1a: Core Replication Pipeline

  • Replication Config + Secondary Connection Pool (WS1) — [replication] TOML config with TopicFilter (regex include/exclude), SASL + TLS support for secondary connections, health probe loop (periodic Metadata requests), config hot-reload on SIGHUP, env var overrides for all replication fields
  • ProduceRequest Interception + Frame Cloning (WS2) — api_key=0 interception in the proxy connection handler; byte-level frame clone with idempotency field stripping (producer_id, epoch, sequence set to -1); per-partition send queues (FIFO, in-order per partition); async fire-and-forget to secondary; <1ms primary path overhead (p99)
  • Circuit Breaker + Metrics + Management API (WS3) — circuit breaker trips after 5 consecutive replication failures (30s backoff with exponential jitter); Prometheus metrics: replication_bytes_total, replication_errors_total, replication_lag_ms, circuit_breaker_state; management API: PUT /replication/{domain}/enable, GET /replication/{domain}/status
