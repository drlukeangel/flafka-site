Schema Impact Analysis

Before you evolve a schema, see exactly what will be affected. Flafka maps every topic, consumer group, Flink SQL statement, and connector that depends on a schema — giving you a complete blast radius visualization.

  • Dependency graph across topics, consumers, and Flink jobs
  • Visual blast radius before any schema change
  • Compatibility checking with conflict detection
  • Schema diff viewer for version comparison
  • Supports Avro, JSON Schema, and Protobuf
Learn more

How blast radius visualization works

When you select a schema for evolution, Flafka scans all topics using that schema, then traces every consumer group, Flink SQL statement, and connector subscribed to those topics. The result is a visual dependency graph showing exactly what will be affected by your change.

Example: You want to remove the legacy_id field from your orders-value schema. Flafka shows you that 3 topics use this schema, 5 consumer groups read from those topics, and 2 Flink SQL statements join on legacy_id. You know exactly what to fix before deploying.

Cost estimation workflow

Before submitting any Flink SQL statement, Flafka estimates the CFU/hour cost based on the query complexity, input topic throughput, and windowing operations. If the estimate exceeds your configured budget threshold, a warning appears with the projected monthly cost.

Permission Tree RBAC

Hierarchical, node-based access control where permissions inherit from parent nodes down the tree. Cluster and environment scoped — each permission tag carries a cluster_id and environment_id so the same role behaves correctly across staging and production clusters. OIDC provider built in: Flafka issues RS256 JWTs with a JWKS endpoint, enabling SSO onboarding and Gateway OAUTHBEARER integration. PCI-DSS Requirement 7 aligned.

  • Hierarchical node tree — permissions inherit from parent nodes
  • Cluster and environment scoping (cluster_id + environment_id on tags)
  • OIDC provider integration — RS256 JWTs, JWKS endpoint, SSO onboarding
  • ACL template presets (Consumer, Producer, Admin)
  • Team Cluster Access Panel — grant or revoke cluster access per team directly in the Permission Tree UI, without touching individual user permissions; cluster_id and environment_id scoping inherited automatically
  • Bootstrap admin with lockout prevention
  • Full audit trail for compliance — PCI-DSS Requirement 7
Learn more

How the Permission Tree works

Permissions are modelled as a tree of nodes. A grant on a parent node automatically propagates to all children — so granting prod access covers every topic, consumer group, and connector under that environment without enumerating each resource. Nodes are tagged with cluster_id and environment_id, making the same permission set work correctly across multiple clusters and environments.

Glob pattern resource matching is still supported for backward compatibility. Examples:

  • analytics-* — All analytics topics
  • prod.* — All production resources
  • team-payments-* — Payment team’s resources

The built-in OIDC provider lets Flafka issue RS256 JWTs directly. Connect your Identity Provider once, and every team member onboards via SSO — no manual credential distribution. The JWKS endpoint enables Gateway OAUTHBEARER mode for end-to-end token-based auth from client to broker.

PCI-DSS Requirement 7 mandates that access to cardholder data is restricted to need-to-know. Flafka’s Permission Tree with audit trails provides the access control evidence auditors require.

Service Accounts (M2M Auth)

Machine-to-machine authentication for CI/CD pipelines, scripts, and Gateway integrations. Service accounts use the OAuth 2.0 client_credentials grant — no human login required. Each account is RBAC-scoped through the Permission Tree, so automated pipelines get the minimum permissions they need and nothing more.

  • OAuth 2.0 client_credentials grant for M2M auth
  • RBAC-scoped via Permission Tree (inherits cluster + environment scoping)
  • API key + secret issuance with one-click secret rotation
  • Gateway SASL integration — service accounts authenticate directly at the broker proxy
  • Audit trail: every service account action logged for compliance
Learn more

CI/CD and automation use cases

Service accounts let your automation infrastructure authenticate to Flafka without storing human credentials in CI/CD secrets. A deploy pipeline needs to create a topic and validate schema compatibility? Create a service account with exactly those permissions, issue a key+secret, and rotate it on a schedule — zero human credential exposure.

Because service accounts are scoped through the Permission Tree, they inherit the same cluster and environment boundaries as human users. A service account granted access to staging cannot accidentally affect production even if its credentials are reused across pipelines.

For Gateway integrations, service accounts support SASL OAUTHBEARER mode — the service account JWT is validated at the Kafka proxy layer, providing end-to-end token-based auth from client to broker without modifying broker configuration.

Why Rust matters for a Kafka proxy

A Kafka gateway sits in the hot path of every produce and consume request. JVM-based gateways introduce latency jitter from garbage collection — a 50ms GC pause on a gateway serving 40k msg/s is a 2,000-message stall. Rust's ownership model eliminates the heap entirely: no GC, no pauses, and memory safety guaranteed at compile time.

Flafka Gateway passes the Kafka wire protocol through with metadata rewriting — your clients connect to the gateway address and see a normal Kafka cluster. Topic routing, partition leadership, and consumer group coordination all work unchanged. No client-side SDK changes required.

Dual-write DR architecture

Flafka Gateway's dual-write mode writes every produced message to two clusters simultaneously — primary and standby — at the proxy layer. No application changes required. The POC achieved 40k msg/s sustained dual-write with zero additional latency overhead. Combined with Flafka's DR Monitoring Dashboard (lag delta, ISR health, failover thresholds), this is an end-to-end active-active DR solution for Kafka without Confluent Cluster Linking costs.

The dual-write architecture is powered by flafka-client — a custom Rust Kafka client library (distinct from rskafka) that connects directly to plain Apache Kafka clusters without requiring Confluent-specific authentication. This is what enables the gateway to write to any Kafka-compatible cluster as a standby target, including self-hosted Apache Kafka, MSK, and Redpanda — not just Confluent Cloud.

Auth integration

Service accounts created in Flafka authenticate directly at the Gateway via SASL OAUTHBEARER Mode 2 — the RS256 JWT issued by Flafka's built-in OIDC provider is validated at the proxy, enforcing Permission Tree RBAC without modifying broker ACLs. One auth system governs both the management UI and the data plane.
